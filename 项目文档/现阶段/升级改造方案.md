## 一、YOLO 模型升级方案

### 1.1 版本选型调研结论

经过对 YOLOv8、YOLO11、YOLO12、YOLO26、YOLO-World 的全面调研：

| 版本 | 发布方 | 发布时间 | 官方权重数 | 任务类型 | mAP@50-95 (M) | CPU推理 | 生产推荐 |
|------|--------|---------|-----------|---------|---------------|---------|---------|
| YOLOv8 | Ultralytics | 2023-01 | 25个 | 5种 | 50.2% | 基准 | 仍支持 |
| YOLO11 | Ultralytics | 2024-10 | 25个 | 5种 | 51.5% | 快15% | 推荐 |
| YOLO26 | Ultralytics | 2026-01 | 30+个 | 6种 | 53.2% | **快43%** | **最新推荐** |
| YOLO12 | 阿里达摩院 | 2025 | 5个 | 仅检测 | 52.5% | 慢 | **不推荐** |
| YOLO-World | 腾讯AI Lab | 2024 | 6个 | 开放词汇 | - | 中 | 辅助兜底 |

**最终选型：YOLO26（最新官方推荐）**

理由：
- Ultralytics 官方最新推荐的生产级模型
- CPU 推理速度比 YOLOv8 快 43%，对 Web 服务响应提升显著
- 端到端无 NMS 设计，部署更简单
- 30+ 官方权重，覆盖 6 种任务（检测/分割/分类/姿态/OBB/语义分割）
- API 与 YOLOv8 完全兼容，代码零修改
- 项目已拥有 `yolo26x.pt` 和 `yolo26n.pt` 权重文件

### 1.2 升级目标

| 维度 | 当前 | 目标 |
|------|------|------|
| 推理框架 | ultralytics >= 8.0.0 | ultralytics >= 8.3.0 |
| 主模型（精细分类） | YOLOv8m (40类, mAP@0.5: 63.44%) | **YOLO26x** (40类, 113MB, 极致精度) |
| 粗分类模型 | YOLOv8l (12类, mAP@0.5: 78.26%) | **YOLO26n** (4大类, 5.5MB, 快速粗筛) |
| 新增兜底层 | 无 | YOLO-World v2 (零样本检测) |

### 1.3 双层架构设计

```
┌─────────────────────────────────────────────────────────┐
│                   Layer 1: YOLO26n 粗分类                 │
│              4大类 (可回收/厨余/有害/其他)                   │
│              5.5MB · <10ms · 高速筛选                      │
├─────────────────────────────────────────────────────────┤
│  置信度 ≥ 0.7 ──→ 路由到 Layer 2 精细分类                  │
│  置信度 < 0.7 ──→ 直接输出大类结果 (低置信度标记)             │
├─────────────────────────────────────────────────────────┤
│                   Layer 2: YOLO26x 精细分类                │
│              40类具体物品                                   │
│              113MB · 高精度识别                              │
├─────────────────────────────────────────────────────────┤
│  置信度 ≥ 0.5 ──→ 输出精细分类结果 + 处理建议                │
│  置信度 < 0.5 ──→ 路由到 Layer 3 兜底                      │
├─────────────────────────────────────────────────────────┤
│                   Layer 3: YOLO-World 零样本兜底            │
│              文本描述检测，无需训练                            │
│              处理训练集外的新型垃圾                            │
└─────────────────────────────────────────────────────────┘
```

### 1.6 YOLO-World 兜底层设计

**原理**：开放词汇目标检测，通过文本描述指定检测类别，无需训练。

```python
from ultralytics import YOLO

model = YOLO("yolo_world_v2_l.pt")
model.set_classes([
    "plastic bottle", "banana peel", "battery", "cardboard box",
    "glass bottle", "aluminum can", "food waste", "styrofoam",
    "cigarette butt", "old clothes", "paper", "metal can"
])
results = model("test_image.jpg")
```

**价值**：
- 识别训练集外的新型垃圾（电子烟、新型包装材料等）
- 无需重新训练即可扩展检测类别
- 作为多模态融合的额外验证视角

### 1.7 YOLO26 核心优势

| 特性 | 说明 |
|------|------|
| **无 NMS** | 原生端到端设计，推理后处理更简单 |
| **CPU 推理快 43%** | 专为边缘/低功耗设备优化，Web 服务响应更快 |
| **移除 DFL 模块** | 简化导出，ONNX/TensorRT 兼容性更好 |
| **MuSGD 优化器** | 融合 SGD + Muon，训练更稳定收敛更快 |
| **小目标增强** | ProgLoss + STAL 损失函数，对细粒度垃圾识别有帮助 |
| **6 种任务** | 检测/实例分割/语义分割/分类/姿态/OBB |
| **API 完全兼容** | `YOLO("yolo26x.pt")` 用法与 YOLOv8 一致 |

### 2.3 全国通用适配设计

面向全国通用，需考虑不同城市的分类标准差异：

| 城市 | 分类体系 | 差异点 |
|------|---------|--------|
| 上海 | 四分类（干/湿/可回收/有害） | "干垃圾"= 其他垃圾，"湿垃圾"= 厨余垃圾 |
| 北京 | 四分类（其他/厨余/可回收/有害） | 名称与国标一致 |
| 深圳 | 四分类 | 与国标一致 |
| 杭州 | 四分类 | 与国标一致 |

## 五、风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| garbage265 下载中断 | 粗分类模型无法训练 | 断点续传，重新运行脚本即可 |
| YOLO26 训练效果不如预期 | 精度不达标 | 可回退到 YOLOv8 + ultralytics 新版框架 |
| 数据库字段重命名导致数据丢失 | 线上数据异常 | 编写迁移脚本，先备份再迁移 |
| 全国城市分类标准差异大 | 逻辑复杂度增加 | 先支持国标四分类，上海等用别名映射 |
| YOLO-World 推理速度慢 | 响应时间增加 | 仅在主模型低置信度时启用，设置超时 |
| 前端全局替换遗漏 | UI 显示不一致 | 替换后全页面截图验证 |
| YOLO26 无 NMS 与现有后处理冲突 | 推理异常 | YOLO26 原生无 NMS，需移除手动 NMS 后处理代码 |
