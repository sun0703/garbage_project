## 一、YOLO 模型升级方案

### 1.1 版本选型调研结论

经过对 YOLOv8、YOLO11、YOLO12、YOLO26、YOLO-World 的全面调研：

| 版本 | 发布方 | 发布时间 | 官方权重数 | 任务类型 | mAP@50-95 (M) | CPU推理 | 生产推荐 |
|------|--------|---------|-----------|---------|---------------|---------|---------|
| YOLOv8 | Ultralytics | 2023-01 | 25个 | 5种 | 50.2% | 基准 | 仍支持 |
| YOLO11 | Ultralytics | 2024-10 | 25个 | 5种 | 51.5% | 快15% | 推荐 |
| YOLO26 | Ultralytics | 2026-01 | 30+个 | 6种 | 53.2% | **快43%** | **最新推荐** |
| YOLO12 | 阿里达摩院 | 2025 | 5个 | 仅检测 | 52.5% | 慢 | **不推荐** |
| YOLO-World | 腾讯AI Lab | 2024 | 6个 | 开放词汇 | - | 中 | 辅助兜底 |

**最终选型：YOLO26（最新官方推荐）**

理由：
- Ultralytics 官方最新推荐的生产级模型
- CPU 推理速度比 YOLOv8 快 43%，对 Web 服务响应提升显著
- 端到端无 NMS 设计，部署更简单
- 30+ 官方权重，覆盖 6 种任务（检测/分割/分类/姿态/OBB/语义分割）
- API 与 YOLOv8 完全兼容，代码零修改

### 1.2 升级目标

| 维度 | 当前 | 目标 |
|------|------|------|
| 推理框架 | ultralytics >= 8.0.0 | ultralytics 8.4.53（已安装） |

### 1.3 双层架构设计

```
Layer 1: YOLO26n 粗分类 → 4大类 (可回收/厨余/有害/其他) · 5.5MB · <10ms
    ↓ 置信度 ≥ 0.7 → Layer 2
Layer 2: YOLO26x 精细分类 → 40类具体物品 · 113MB · 高精度
    ↓ 置信度 < 0.5 → Layer 3
Layer 3: YOLO-World 零样本兜底 → 文本描述检测，无需训练
```

### 🛡️ 安全防护机制详情

#### 新增文件：train_yolo26_safe.py

**核心改进：**

1. **SystemMonitor 类（系统资源监控器）**
   - 后台线程持续监控系统状态
   - 内存使用率监控（psutil库，每30秒检查一次）
   - GPU温度和显存监控（pynvml库，每10秒检查一次）
   - 自动记录到独立日志文件 `training_monitor.log`

2. **动态资源调整**
   ```python
   SAFETY_CONFIG = {
       "memory_warning_threshold": 85,      # 内存警告阈值 (%)
       "memory_critical_threshold": 92,     # 内存紧急阈值 (%)
       "gpu_temp_warning": 80,              # GPU温度警告 (°C)
       "gpu_temp_critical": 95,             # GPU温度紧急 (°C)
       "save_period": 5,                    # checkpoint保存间隔（epoch数）
       "safe_batch_size_x": 4,              # YOLO26x 安全batch大小
       "safe_workers": 2,                   # 数据加载线程数
   }
   ```

3. **频繁Checkpoint保存**
   - 原设置：每个 epoch 结束后保存 last.pt
   - 新设置：`save_period=5`（每5个epoch额外保存一次完整checkpoint）
   - 最大损失：即使崩溃，最多只损失4个epoch的训练进度

4. **异常捕获与恢复机制**
   - try-except 包装整个训练流程
   - KeyboardInterrupt（Ctrl+C）友好处理
   - 崩溃后自动检测可用的 last.pt checkpoint
   - 下次启动时可选择断点续传

5. **详细日志记录**
   - 文件日志：`training_monitor.log`（INFO级别以上全部记录）
   - 控制台日志：WARNING级别以上显示
   - 时间戳精确到秒
   - 包含异常堆栈跟踪

**监控线程架构：**

```
主进程（训练）
    │
    ├── MemoryMonitor 线程（守护线程）
    │   └── 每30秒检查：
    │       ├── mem.percent > 85% → WARNING 日志
    │       └── mem.percent > 92% → CRITICAL 日志 + 标记紧急标志
    │
    └── GPUMonitor 线程（守护线程）
        └── 每10秒检查：
            ├── temp > 80°C → WARNING 日志
            ├── temp > 95°C → CRITICAL 日志 + 标记紧急标志
            └── 记录 VRAM 使用情况
```
